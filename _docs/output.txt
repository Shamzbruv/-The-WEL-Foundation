Revamp Research Report for The WEL Foundation
Website
Executive summary
The current public website for The WEL Foundation is a small Wix-style brochure site with core pages for
Home, About, Programs, and Contact, plus a publicly accessible  Groups feature. The front-facing pages
communicate mission and basic program categories, but they are extremely light on program detail and
“next-step” guidance, and the Contact experience appears minimal. 
The most urgent, high-impact issue is that the Groups section is public and contains content that appears
unrelated to the organization (e.g., sports betting spam). This directly undermines trust and can create
reputational and SEO risk because search engines can index these pages. 
Based on the client questionnaire (attached CSV) and the current site content, the revamp should prioritize:
a  full  brand  refresh;  clearer  service/program  explanations  (including  the  newly  added  SUD  program);
improved and accessible intake/referral forms with secure file uploads; and an employee login + dashboard
workflow so staff can receive, manage, and follow up on submissions (including reminders to process
submissions within 3 business days). (Client questionnaire, provided file)
A  feasible  delivery  approach—given  the  client’s  requested  completion  date  of  April  3,  2026 (next  day
relative to today)—is a phased plan: - Phase A (Immediate risk reduction / stabilization): remove or lock
down Groups; clean indexability; publish a simplified, trustworthy “MVP” site with clear programs, CTAs, and
a working Contact + basic forms. -  Phase B (Full rebuild): brand refresh, complete rewrite, structured
program pages, event calendar/resources, and the secure staff portal with reminders and document upload
workflow. 
Estimated effort depends on scope decisions (especially around HIPAA sensitivity and portal complexity),
but a typical range is: - Phase A: 20–40 hours - Phase B: 120–220 hours (includes auth, RLS, storage, staff
dashboard, reminder automation, content rewrite coordination, QA, and launch hardening).

Baltimore community and lists a long set of skills/support areas (e.g., conflict resolution, anger

not show form fields; treat as “form present but not auditable from extracted content.”

Critical risk: public “Groups” section appears compromised by spam
The Groups section is public and includes posts with content unrelated to the organization (example: “Guide
to read odds in Indonesia football betting” with outbound links). This is a high-severity trust issue for
prospective clients and referral partners and can pollute search results for the brand.

UI/UX recommendations
Replace the current “thin” Programs listing (headings only) with scannable summaries and clear next
steps; the existing Programs page provides only headings and Get in Touch links. 
Remove or replace the public Groups feature entirely; it is not aligned with the organization’s service
mission and is currently showing spam. 
Use a consistent CTA hierarchy:
Primary: Start Intake, Refer a Client
Secondary: Programs, Contact Us
Tertiary: Events, Resources, Learn More
Create “trust scaffolding” everywhere sensitive info is requested: what data is collected, why, how it’s
protected, and expected response time.
Ensure mobile-first spacing and readable type scaling; responsive design is strongly recommended
for mobile-first indexing. 
Accessibility baseline requirements
Given the heavy form emphasis, WCAG form guidance is critical: - Provide explicit labels/instructions for
each  input  and  describe  errors  in  text  (WCAG  2.2  input  assistance  criteria).   -  Ensure  keyboard
accessibility, clear focus states, and logical heading order . - Use error summaries at top of step pages and
inline field-level errors. - Avoid “placeholder-only” labeling. - Use sufficient contrast for blue/yellow branding
and test with automated + manual checks. - Provide accessible authentication patterns (avoid CAPTCHAs
without accessible alternatives), especially if combined with secure workflows. 
SEO basics aligned to this project
Maintain clean titles and headings (Google considers <title> and other on-page signals for title
links). 
Ensure responsive design and parity of content between mobile and desktop for mobile-first
indexing. 
Publish an XML sitemap and keep it accurate.

Technical architecture
Frontend implementation approach
Recommended build model - Static-first site: hand-authored pages compiled from templates into static
HTML (easy CDN hosting, low attack surface). - Use a simple templating build tool (e.g., 11ty/Astro) to avoid
repeating header/footer and to keep page creation consistent; output remains plain HTML. - Vanilla CSS
with design tokens (CSS variables) for color , spacing, typography. Use modern layout (Flexbox/Grid). - Vanilla
JS  modules  for  progressive  enhancement:  form  wizard  behaviors,  mobile  menu,  light  UI  interactions,
Supabase auth state.
Responsive breakpoints (suggested) - 0–479px: small mobile - 480–767px: large mobile - 768–1023px:
tablet - 1024–1279px: desktop - 1280px+: large desktop
Progressive enhancement - Marketing pages work without JS. - Contact/Intake/Referral forms can be JS-
enhanced, but for reliability and security, prefer posting to a serverless endpoint (Netlify/Vercel/Cloudflare)
that writes to Supabase with the service role key (kept server-side). 
When to use additional tech - If the staff dashboard grows into a complex app (filtering, attachments,
assignment  workflows),  consider  a  small  React/Vite  app  only  under  /staff/* while  the  public  site
remains static. This keeps complexity contained and avoids framework overhead sitewide.
Supabase backend design
This section uses Supabase as the primary backend for authentication, roles, form submissions, and file
storage.
Core Supabase components to use
Auth for staff login.
Postgres + Row Level Security (RLS) for data protection; requests map to anon and 
authenticated Postgres roles. 
Storage for secure document uploads (private buckets + RLS and/or signed URLs). 
Edge Functions + Cron for reminders and automated workflows:
Supabase supports pg_cron scheduling and invoking Edge Functions via pg_net. 
Vault/Secrets pattern for storing tokens for cron-driven function calls (per Supabase scheduling
guidance). 
Never expose the service role key in frontend code; use it only in server-side functions.

Authentication and user roles
For role-based access control: - Use Supabase RBAC guidance via  custom claims and a  Custom Access
Token Auth Hook to attach user_role to the JWT, then enforce in RLS policies. 
Suggested  roles  (adjustable):  -  admin (full  access)  -  staff (view/manage  submissions)  -
intake_coordinator (optional specialized staff function)
Contact forms, intake/referral forms, and reminders
Contact form (low sensitivity) - Store into  contact_messages table. - Allow inserts from  anon with
strong validation, spam protection (rate limiting, CAPTCHA/Turnstile), and minimal fields.
Intake/referral forms (high sensitivity) - Store into submissions table (type: intake or referral)
with a status workflow: - new → in_review → complete (or needs_more_info) - File uploads: - Use
a private Storage bucket; access controlled via RLS and/or signed URLs.  - Reminder automation: - On
submission creation, compute due_at. - Scheduled function runs daily to find overdue submissions and
email assigned staff; log reminders. Supabase supports cron scheduling in Postgres and invoking Edge
Functions. 
HIPAA/compliance note (must be treated as a project risk) Because the client wants HIPAA-conscious
handling and the forms include ID/insurance/supporting documents (potentially PHI), vendor selection and
contracts matter . HHS notes that entities handling PHI on behalf of covered entities may be “business
associates” and typically require appropriate agreements (BAAs) and safeguards. 
This report can design secure controls, but compliance determination requires legal and vendor contracting
review.

This example demonstrates a “progressive enhancement” approach: a standard HTML form with JS that
inserts into Supabase. For sensitive submissions, route through a serverless endpoint so the service role key
stays server-side, per Supabase security guidance.

<labelfor="phone">Phone (optional)</label>
<inputid="phone"name="phone"autocomplete="tel"/>
<labelfor="message">How can we help?</label>
<textareaid="message"name="message"rows="5"required></textarea>

functionsetStatus(msg) { statusEl.textContent= msg; }
form.addEventListener("submit", async(e) =>{
e.preventDefault();
setStatus("");
// Basic client-side validation (also enforce server-side).
constname= form.name.value.trim();
constemail= form.email.value.trim();
constphone= form.phone.value.trim();
constmessage= form.message.value.trim();
if(!name||!email||!message) {
setStatus("Please complete all required fields.");
return;
}
// Optionally associate logged-in user (if using auth for public visitors

const{ data: userResp} = awaitsupabase.auth.getUser();
